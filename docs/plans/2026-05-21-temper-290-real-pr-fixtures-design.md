# Design ‚Äî #290: Real-PR-Derived Eval Fixtures + Harness Synthetic/Real Split

## Problem

The 10 synthetic fixtures in `skills/temper/evals/evals.json` (`1a`, `1b`, `2`, `2-reattrib`, `3`, `4`, `5`, `6-tenancy-forged-callback`, `7-rollback-orphan-fk`, `8-defense-in-depth-not-slop`) are short (‚â§70 LOC each), single-file, and scrubbed clean of contextual noise. They reliably exercise the lens vocabulary but cannot catch reviewer drift on realistic-noise diffs ‚Äî i.e., a reviewer that "sees" the lens pattern only when the diff is conspicuously stripped of distractors. Innovate Angle 5 from #287 surfaced this; #290 was filed to close it.

## Goal

Add ~5 real-PR-derived fixtures alongside the synthetic ones, one per Targeted Lens (Surgical, DRY, SRP, OCP) plus one mixed-lens case. Each fixture embeds an actual merged-PR diff (or a slice of one) with a plausibly-natural lens violation, exercising the reviewer against realistic noise (full headers, surrounding context, real variable names). Extend `run_evals.py` to report synthetic vs real pass-rates separately so future drift can be attributed.

**Out of scope:** Real-PR fixtures for the two Categories (Tenancy, Rollback). Those landed in PR #293 only hours ago; their synthetic fixtures are still proving out the parser. Defer to a follow-up after the Lens real-PR pattern beds in.

All existing 10 fixtures gain `source: "synthetic"` (explicit, not implicit). (See Non-Goals M3 fallback for the `gap_documented: true` carve-out when a lens column genuinely lacks any band-landing injection site after K=3 re-author attempts.)
2. Each real-pr fixture cites the source PR number + commit SHA in a new `source_pr` field; fixture diff is a verifiable slice (subset of lines) of that PR's diff, with one injected lens violation if the source PR did not naturally contain one.
3. The path `skills/temper/lens_runner.py` does NOT exist; the real path is `skills/temper/evals/lens_runner.py`).
4. `run_evals.py` exposes `--source synthetic|real-pr|all` filter (default `all`); summary output groups pass-rate by source AND emits a per-lens Drift Delta line computed from per-trial rates (not binary verdicts).
5. `last_run.json` segregates synthetic vs real fixture results into `by_source: {...}` and adds `drift_delta: {<lens>: <float>}` derived from `per_trial_verdicts` rates. `last_run.json` is the volatile most-recent-run snapshot and is **gitignored**; the comparison baseline lives in a separate `evals/baseline.json` that IS committed (see Harness ¬ß8).
6. **Baseline-capture acceptance (not all-PASS).** At land time, the run's `drift_delta` and `by_source` numbers are captured to `evals/baseline.json` via the full 3-step protocol with a **date-stamped run_id** to avoid `FileExistsError` on rebaseline (S1): `python run_evals.py stage R-baseline-<UTC-yyyyMMdd> --source all` ‚Üí `/temper-eval-collect R-baseline-<UTC-yyyyMMdd>` ‚Üí `python run_evals.py score R-baseline-<UTC-yyyyMMdd> --write-baseline` (see "Harness invocation model (post-#297)"), and that file is committed as the baseline. Acceptance is that the baseline is *recorded*, NOT that delta == 0 or that all real-PR fixtures PASS. Drift detection becomes a comparison on *future* runs: a per-lens delta swing > 1/5 (i.e., a per-trial-rate change > 0.2 in either direction) against the committed baseline flags suspected reviewer drift. This resolves the AC1‚ÜîAC6 contradiction: real-PR fixtures are allowed to fail at land time ‚Äî that failure becomes part of the baseline and the signal the issue exists to surface.
7. Drift check (whatever guards `temper-reviewer.md` / `shared/reviewer-common.md` consistency) is unchanged ‚Äî no edits to `temper-reviewer.md` or `shared/reviewer-common.md`. The `_synth_plan_reference` data-plumbing change in `run_evals.py` (switching the source field read from `expected_output` to `pr_description`) IS in scope but does not touch reviewer-prompt template content ‚Äî it only changes which fixture field gets injected into the existing template slot.
AC8 (split into 8a + 8b for distinct calibration concerns):

8a. **Informative-band hurdle for real-PR fixtures (S1; band scales with n=trials).** At land time, each real-PR fixture's per-trial PASS rate from the baseline-capture run must fall in the **informative band `(0, 1.0)` exclusive at n=trials** ‚Äî NOT 0/n (uniformly failing ‚Äî likely unscoreable injection or malformed prompt) AND NOT n/n (uniformly passing ‚Äî trivially easy, no drift discrimination power). With the mandatory `trials=5` for lens-column-participating fixtures (Harness ¬ß2(i)), this resolves to `1/5‚Äì4/5`. The band is expressed parametrically so it scales correctly if `trials` is ever raised (e.g., per SP1 follow-up, trials=10 ‚Üí band `1/10‚Äì9/10`). Fixtures landing at the 0/n or n/n extremes are re-authored before the baseline is committed. **Carve-out (S4 enumeration, not prose):** a real-PR fixture may land at n/n ONLY if it carries an explicit `counter-rule: true` flag on the fixture object AND a written rationale in `expected_output` describing why uniform pass is the expected lens-validation signal (mirrors the fixture-8 defense-in-depth shape). No `counter-rule: false`-equivalent escape hatch exists for n/n absent the flag, and **no equivalent exception exists for 0/n** ‚Äî uniformly-failing fixtures are always re-authored.

**Baseline-iterations cost (SP1).** Because the band check is applied to the same n=5 baseline-write run that becomes the committed baseline, a fixture with true rate ‚âà 0.7 has roughly 1-in-3 chance of producing 5/5 on any given baseline write, requiring re-authoring and a fresh stage‚Üícollect‚Üíscore cycle. Expected baseline-write iterations across 5 real-PR fixtures is materially > 1 (estimated 1.5‚Äì2 per fixture; aggregate 8‚Äì10 cycles in the worst case). This cost is in ADDITION to the synthetic-split calibration (Test Plan ¬ß7) and MUST be reserved in the implementer's time/dispatch budget. Follow-up consideration: raising real-PR trials to 10 widens the band to `1/10‚Äì9/10` and materially lowers the rejection-sampling rate; deferred to a follow-up once the real-PR pattern beds in.

8b. **Signal-existence check at baseline-write time (S2; noise-calibrated).** Baseline-capture acceptance includes a signal-existence gate. The naive "|delta(C)| > 0.3 on ‚â•1 of 4 columns" formulation at n=5 has roughly 80% false-positive rate from sampling noise alone (per-rate binomial stdev ‚âà 0.224 at n=5, single-run delta stdev ‚âà 0.317 ‚Üí ~33% chance per column of tripping a 0.3 threshold under zero true signal; 1 ‚àí 0.67‚Å¥ ‚âà 0.80 across 4 columns). The gate is therefore **noise-calibrated** in one of two operating modes (implementer picks one and documents it in the PR description):

- **Mode A (raised threshold at n=5):** require `|delta(C)| > 0.5` on at least 1 column AND `|delta(C)| > 0.3` on a second column. The 0.5 single-column trip is ‚âà 1.6œÉ on single-run delta stdev (much harder to reach by noise); the second-column 0.3 confirmation drops the joint FPR materially. **(R4 S4) Joint-FPR derivation.** Per-column FPR at threshold 0.5 with n=5 binomial single-run delta stdev (‚âà 0.317) is ‚âà 6.25% (single-column trip at ~1.6œÉ). Per-column FPR at threshold 0.3 is ‚âà 33% (single-column trip at ~0.9œÉ). Under an *independence* assumption across columns, the joint FPR for tripping 0.5 on column A AND 0.3 on any other column is approximately `0.0625 √ó (1 ‚àí 0.67^3) ‚âà 0.0625 √ó 0.70 ‚âà 4.4%` (‚âà 0.625% per ordered pair √ó ~7 pair orderings); the simpler "pick one trip column then any other confirm" estimate is ‚âà 0.625%. **Independence assumption stated:** lens columns share a reviewer dispatch but use different fixtures; rates are weakly correlated through reviewer-noise (temperature jitter, cache state). The independent-binomial 0.625% is an UPPER bound on independent failures only; correlated failures inflate the actual joint FPR somewhat. **Empirical calibration is required** (per Test Plan ¬ß7's calibration cycle) to confirm the joint FPR stays below 5% in practice; if calibration measures joint FPR > 5%, the design bounces back to revisit the 0.3 confirm threshold (likely raise to 0.4).
- **Mode B (raised n for baseline runs, R4 S2 ‚Äî operated via `--trials-override`):** Mode B is operated by passing `--trials-override 10` at the `stage` subcommand. The fixtures' `replicate_rule.trials` stays at `5` in `evals.json`; gate (i) reads the unchanged static `trials: 5` field at load time and is satisfied (the validator never sees the runtime override flag). The harness then uses 10 trials at dispatch time, and AC8a's informative-band re-derivation at n=10 (`1/10‚Äì9/10`) applies to the resulting rates. Cost: ~50 extra reviewer dispatches, one-time per rebaseline. **R4 S2 correction.** Earlier drafts of Mode B claimed gate-(i) cleanliness "because the override value is in `{5, 10}`" ‚Äî that claim was vacuous because gate (i) reads the static `replicate_rule.trials` JSON field, not the runtime override. The correct framing is that gate (i) is *orthogonal* to `--trials-override`: gate (i) cares only about the static field; Mode B operates entirely at dispatch time and bypasses gate (i)'s relevance. If a maintainer wanted Mode B as the permanent posture (`trials: 10` on every run), they would edit `evals.json` directly ‚Äî and gate (i) would still pass because `10` is in the accepted set. Pick ONE path per rebaseline; document the chosen path in the PR description. Single-run delta stdev at n=10 shrinks to ‚àö2 √ó ‚àö(0.25/10) ‚âà 0.224 and a `|delta(C)| > 0.3` single-column threshold becomes ‚âà 1.3œÉ, with per-column false-positive rate ‚âà 19%, joint-across-4-columns FPR ‚âà 57% ‚Äî still too noisy. Mode B therefore also uses `|delta(C)| > 0.4` at n=10 (‚âà 1.8œÉ; per-column FPR ‚âà 7%, joint ‚âà 25%) as its single-column trip.

The chosen threshold + n is documented in the same prose-style as the Risk ‚Üí Noise Floor calibration (numeric stdev derivation, claimed FPR, acknowledged residual noise). The "bounces back to design" consequence is preserved ‚Äî but only after the calibrated threshold confirms genuine null-signal, not from a noise-floor trip. If all column deltas land within the calibrated band, the calibration is treated as "no signal" ‚Äî likely cause is that synthetic and real fixtures are equally hard / equally easy, or the methodology is invalidated. This prevents committing a degenerate baseline that locks in ongoing maintenance cost (Test Plan ¬ß8 rebaseline overhead) with zero offsetting signal value.

9. **`pr_description` is scope-only for ALL fixtures.** Every fixture (synthetic AND real-PR) has a `pr_description` field that describes the PR's surface intent only ‚Äî no mention of the lens violation, the expected finding, or any rationale for why the fixture should pass/fail. The existing `expected_output` text on the 10 synthetic fixtures must be SPLIT by hand: the scope-prose half ‚Üí `pr_description`; the rationale/expectations half ‚Üí `expected_output` (which becomes maintainer-readable documentation, no longer consumed by the reviewer prompt). Bulk-copying `expected_output` into `pr_description` is explicitly disallowed ‚Äî the implementer hand-edits each of the 10 existing fixtures.

## Harness invocation model (post-#297)

**The harness is no longer a monolithic `run_evals.py` invocation.** Per #297, `run_evals.py` is a subcommand dispatcher with two leaf commands (`stage`, `score`), and reviewer dispatch happens out-of-process via `/temper-eval-collect` (per `feedback_no_claude_p` ‚Äî `claude -p` is banned in this repo). Every step in this design that previously read as a single command MUST be expanded to the 3-step protocol:

1. `python run_evals.py stage <run_id> --source <synthetic|real-pr|all>` ‚Äî renders fixtures √ó trials into per-trial dispatch files under the run directory. (The `--source` filter lives ONLY on `stage`.)
2. `/temper-eval-collect <run_id>` ‚Äî human-triggered subagent dispatch that invokes the reviewer on each staged file and writes outputs back. **This is the dispatch boundary; the harness does NOT call the reviewer in-process.**
3. `python run_evals.py score <run_id> [--write-baseline | --compare-baseline]` ‚Äî aggregates collected outputs, writes `last_run.json`, and (optionally) writes/compares `evals/baseline.json`. (Both baseline flags live ONLY on `score`.)

**Mapping of design steps to subcommands:**

| Design item | Subcommand | Notes |
|---|---|---|
| AC4 `--source` filter | `stage --source` | Already shipped (line 790). Fixture-selection filter applied at stage time. |
| AC6 baseline capture | `score --write-baseline` (after stage+collect) | Writes `_BASELINE_PATH` (hardcoded `evals/baseline.json`); not a configurable path. |

This is operational reality, not optional friction; the PR description / runbook MUST document the sequence so future maintainers can rebaseline reliably.

**Run-id convention (S1).** All baseline-capture and rebaseline invocations use **date-stamped** run_ids of the form `R-baseline-<UTC-yyyyMMdd>` (e.g., `R-baseline-20260524`). Re-using the bare `R-baseline` will raise `FileExistsError` from `stage()` (see `run_evals.py:216-219`) on the second rebaseline unless `--force` is passed. Date-stamping is the canonical convention because it preserves per-rebaseline dispatch artifacts under `evals/runs/` for post-hoc inspection; `--force` is the alternative if disk pressure requires reclaiming prior runs. The Test Plan and protocol examples use date-stamped IDs throughout.

## Sourcing Strategy

**Hybrid: inject into real-PR slices, not pure-mine.** Pure-mining merged PRs for naturally-occurring lens violations is low-yield ‚Äî merged PRs were generally clean (lens violations would have been review-blocked). Pure synthetic injection ignores realistic noise. The hybrid:

1. Pick 5 merged Crucible PRs (recon shortlist: #248, #247, #245, #213, #261 ‚Äî the latter being the temper rename PR that landed SHA `056176e`).
2. For each, take a slice of the actual diff (10‚Äì80 LOC contiguous from a real file). **Soft cap: ~80 LOC of diff content per fixture.** Validation emits a warning (not error) if `len(fixture['prompt'].splitlines()) > 100`; values in the 80‚Äì100 range are acceptable when the realism criterion requires it (e.g., a real diff hunk needs surrounding context to read naturally). Hard upper bound for the warning-to-error escalation is 100 LOC. **Tradeoff:** softer cap admits larger reviewer-prompt budgets per fixture (cost: ~5‚Äì10√ó synthetic prompt size, see Risk ‚Üí cost) in exchange for realistic-noise fidelity; enforcing as a hard error would force authors to truncate slices below their natural reading unit.
3. Inject a lens-violating edit at a plausibly-natural site (e.g., add a new `elif` arm where a registry already exists in the same module ‚Üí OCP). **Plausibility criterion (operational, AND-tightened per SP2):** the injection site must satisfy **BOTH** of the following (the prior OR formulation was structurally weak ‚Äî every author would take the cheap OR-escape):
   - (criterion A) the injection LOC is ‚â•20 LOC from the diff's first added line within the slice, AND the injection LOC is ‚â•10 LOC into the slice's content from the slice start (i.e., not at the literal first line of the slice ‚Äî distance measured against the unified-diff line numbering within the chosen slice, not against the parent PR's whole diff);
   - (criterion B) the fixture includes ‚â•1 unchanged file shown for context (matching existing fixture 4's structure), AND that unchanged-context file contains ‚â•30 LOC of content (no padding with a 3-line stub).
   Fixtures failing either criterion are rejected as "synthetic-with-extra-noise." Note: this is a tightening from R1 (which allowed an OR escape); authors who previously planned an OR-only fixture must rework to satisfy both criteria. The cost is real (more authoring time per fixture) but is the price of the realistic-noise value proposition holding up.
4. Document the injection site + rationale in the fixture's `expected_output` prose **for human/maintainer reference only** ‚Äî this field is no longer consumed by the reviewer prompt (see Harness Changes ¬ß1).
5. Author a separate `pr_description` field that the reviewer prompt consumes as the "What was requested" scope statement. This field must NOT mention the injected lens violation or its rationale ‚Äî it describes the PR's surface intent only.

Each fixture's `source_pr` field cites the PR + SHA so future maintainers can verify the diff origin.

**Authoring guidance for the synthetic-fixture `pr_description` split (R3 SP2, SP3):**

- **Fixture `5` (negative / zero-findings sentinel).** The split must NOT prime the reviewer toward "no findings." Use a neutral one-sentence template that states the surface scope. Example: *"Add `slugify_with_max_length` helper that wraps the existing `slugify` utility."* No "this is correct," "no issues expected," or "PR is clean" language.
- **Fixture `8` (defense-in-depth counter-rule).** This fixture's expected permissive verdict is structural to its design ‚Äî the "permissive on drive-by" rationale is *the* reason the fixture exists. Acceptable carve-out: the `pr_description` documents the surface intent only (e.g., *"Reformat the RLS-check block and rename a typo'd variable in the same file."*); the permissive-verdict rationale stays in `expected_output` as documentation. Documented here explicitly to pre-empt reviewer confusion about why fixture `8` gets a more-permissive `pr_description` partition than the others.

**Mixed-lens fixture (5th):** Carry a diff that triggers two lenses (e.g., DRY + SRP, or Surgical + OCP). The expectation set asserts both fire and neither suppresses the other. For Drift Delta accounting, the mixed fixture **contributes to BOTH lens columns** (one per lens it triggers). Double-counting is acceptable because per-lens delta is informational, not summed into a single headline number; the mixed fixture's `lens_under_test` is a list of the two lenses it pairs against.

## Storage Strategy

**Single manifest, tagged.** No new subdirectory. Add `source: "synthetic" | "real-pr"` to every fixture object in `evals.json`. Rationale: keeps the harness loader unchanged (single-file glob is fine), avoids cross-file dedupe concerns, makes diffs reviewable in one place. The 10 existing fixtures gain `source: "synthetic"`; new fixtures gain `source: "real-pr"`.

Trade-off considered: separate `real-pr/` subdir with one file per fixture. Pro: smaller diffs per-fixture, easier to add without touching the master manifest. Con: parser must walk a glob, schema duplicated per file, source-grouping requires aggregation. **Reject** ‚Äî single manifest is simpler for the current scale (‚â§20 fixtures).

6. **Drift Delta block (innovate addition, per-trial granularity; F2 ‚Äî driven by `lens_column`).** A summary line per-lens computed from **per-trial pass rates**, not binary aggregate verdicts. **Three-state per-trial outcome enum (R3 SP1):** each trial is recorded as one of `PASS | FAIL | ERROR`. **Definition ‚Äî "PASS trial":** a single trial is PASS for a given fixture iff every expectation's mechanical check returns Pass for that trial. **Definition ‚Äî "FAIL trial":** the reviewer dispatched, output parsed cleanly, but at least one expectation's mechanical check did not return Pass. **Definition ‚Äî "ERROR trial":** the trial failed at dispatch or parse time (subprocess crash, malformed reviewer output, timeout) ‚Äî distinct from a substantive FAIL. **Rate formula (R3 SP1):** ERROR trials are excluded from the denominator. Rates computed as `PASS / (PASS + FAIL)`. **Column grouping rule (F2):** for each Drift Delta column `C ‚àà {Surgical, DRY, SRP, OCP}`, gather all fixtures whose `lens_column` contains `C` (singleton string or list ‚Äî both forms covered). Then:
   - `synthetic_pass_rate(C) = (Œ£_{f ‚àà column C, f.source == "synthetic"} PASS_trials(f)) / (Œ£_{f ‚àà column C, f.source == "synthetic"} [PASS_trials(f) + FAIL_trials(f)])`
   - `real_pass_rate(C) = (Œ£_{f ‚àà column C, f.source == "real-pr"} PASS_trials(f)) / (Œ£_{f ‚àà column C, f.source == "real-pr"} [PASS_trials(f) + FAIL_trials(f)])`
   - `delta(C) = synthetic_pass_rate(C) ‚àí real_pass_rate(C)`

   **Rationale for ERROR exclusion (R3 SP1).** R2's "errors counted as FAIL" rule silently baked dispatch noise into the committed baseline as systematic bias (e.g., 1-of-25 dispatch failures on baseline day ‚Üí +4% per-rate distortion that compounds across rebaselines). Promoting ERROR to a first-class outcome and excluding from denominator preserves baseline unbiasedness; gate (l) below makes ERROR-rate spikes fail-loud rather than silently distort the baseline.

The score summary surfaces a WARNING line enumerating gap-documented columns so future rebaseline runs do not silently treat the gap as a numeric `0.0` drift. Cross-lens aggregates (any future "max drift" or "mean drift" summaries) skip `n/a` columns rather than coercing them to 0.

This is the literal staging behavior at `run_evals.py:233-234` (each fixture's `rule.get("trials", 1)`). Harness ¬ß2(i) constrains column-participating fixtures to `trials=5` so the sum is equivalent to `5 √ó count` in practice ‚Äî but the formula is written per-fixture to remain correct if ¬ß2(i) is ever relaxed and to match implementation reality.

   With trials=5 this gives 11 buckets per fixture (0/5, ‚Ä¶, 5/5) and finer-grained deltas at the column level. Example: `Drift Delta (per-trial rates) ‚Äî Surgical: 0.00 ¬∑ DRY: +0.20 ¬∑ SRP: +0.40 ¬∑ OCP: 0.00`. **Mixed-fixture rule (no double-count on synth side):** because mixing is achieved by listing two `lens_column` values on the *real-PR* mixed fixture (not by adding a mixed fixture to the synthetic side), the synthetic denominator for each column is determined purely by `{f : f.source == "synthetic" ‚àß C ‚àà f.lens_column}`. 7. **Baseline + tolerance.** A committed `evals/baseline.json` is the drift-comparison baseline (see Harness ¬ß8 for file separation). On a subsequent run, `python run_evals.py score <run_id> --compare-baseline` (boolean flag, reads `_BASELINE_PATH = _EVALS_DIR / "baseline.json"` ‚Äî hardcoded, not a positional path) loads the prior baseline and emits a drift flag for any lens whose `delta` changed by more than **0.7** against the baseline. **Tolerance rationale (R3 F1 option (a) ‚Äî corrected math).** The gate measures *swing-vs-baseline*, i.e., the difference of two deltas computed across independent runs. Each delta is itself a difference of two per-trial rates (synthetic vs real-pr). For a lens column with 1 synthetic-twin fixture and 1 real-PR fixture at n=5 trials each (the worst case among current columns ‚Äî SRP and OCP), the per-rate binomial stdev is ‚àö(0.25/5) ‚âà 0.224; a single-run delta is the difference of two such rates ‚Üí stdev ‚àö2 √ó 0.224 ‚âà 0.317; and the swing (delta now minus delta at baseline) is the difference of two single-run deltas ‚Üí stdev ‚àö2 √ó 0.317 ‚âà **0.447**. (Lens columns with 2 synthetic twins ‚Äî Surgical, DRY ‚Äî have lower swing stdev because their synthetic denominator doubles; 0.447 is the worst-case bound that the gate must respect.) A 0.7 tolerance ‚âà 1.5œÉ of swing stdev on the worst-case columns. **False-positive rate (honest hedge, S2).** The "‚âà 12% per lens, ‚âà 40% per-run-any-lens" estimate that earlier drafts cited assumes (a) independence across the synthetic and real arms within a run, and (b) independence across lens columns and across runs. Both assumptions are wrong: within-run shared reviewer state (temperature jitter, prompt-cache warm/cold, sampling drift) positively correlates the synth and real arms (lowering single-run delta stdev); across-run correlation is weaker but nonzero (different cache state, OS/network noise). The net effect on FPR is **unknown without empirical calibration** ‚Äî the actual rate may be materially above or below the independent-binomial estimate. The 0.7 swing tolerance is chosen as a heuristic "loud trip-wire" ‚Äî loud enough to fire on visible reviewer drift, quiet enough to not fire every run on n=5 fixtures. **Follow-up calibration task (to be filed as an issue):** after the baseline is committed, run the eval N=10 times against the captured baseline and measure the actual swing distribution per lens column; use the empirical distribution to revise the tolerance and report a measured FPR. Until that task lands, all FPR numbers in this design are best-effort estimates, not measured rates. **Tradeoff:** real reviewer drift producing a 0.4‚Äì0.7 swing will be missed at n=5; FPR remains material. **Future work:** consider raising `trials` to 10 specifically for real-PR fixtures + their synthetic twins (‚âà 25 extra reviewer dispatches per run) once the real-PR pattern beds in, which would let the tolerance tighten to ‚âà 0.5 (see Risk ‚Üí Noise Floor).

- **Injection plausibility.** A poorly-injected violation (one that no human would actually write) regresses to synthetic-with-extra-noise and adds zero signal. Mitigation: operational plausibility criterion in Sourcing Strategy ¬ß3 (‚â•20 LOC distance or unchanged-file context); maintainer rationale recorded in `expected_output` (documentation-only, not reviewer-visible ‚Äî see Harness ¬ß1); Phase 3 code review explicitly checks fixtures against the criterion.
- **Source PR linkage rot.** If a referenced PR is force-pushed or its branch deleted, the `source_pr` cite becomes dead. Mitigation: cite commit SHA, not branch ‚Äî SHAs are immutable in the merged-to-main tree. (M1) `source_pr` format is constrained to `^#\d+ @ [0-9a-f]{7,40}$` ‚Äî see gate (b) in Harness ¬ß2, which IS the load-time validation for this format.
- **Replicate-rule cost + prompt inflation.** Each fixture runs `trials` √ó reviewer dispatches. 5 new fixtures √ó 5 trials = 25 reviewer runs per eval invocation. Real-PR slices are also larger (5‚Äì10√ó prompt size vs synthetic). Mitigation: hard 80 LOC diff cap per fixture (Sourcing Strategy ¬ß2); keep `trials: 5, threshold: 3` for real-pr (same as synthetic); do not over-replicate.
- **Non-deterministic dispatch.** `claude -p` is non-deterministic; a single-snapshot delta is sampling noise. Mitigation: baseline-comparison mechanism (Harness ¬ß7) ‚Äî drift is signaled only when delta swings >0.7 vs the committed baseline, not from a single absolute value (distinct from AC8b Mode B's 0.4 two-column threshold, which gates per-column negative-delta warnings, not full-suite drift).
- **Noise floor (R3 F1, corrected math).** The gate measures swing-vs-baseline (a difference of two single-run deltas, each itself a difference of two per-trial rates). At n=5 trials per fixture, worst-case swing stdev (lens columns with 1 synthetic twin) is ‚àö2 √ó ‚àö2 √ó ‚àö(0.25/5) ‚âà 0.447. The chosen 0.7 swing tolerance sits at ‚âà 1.5œÉ on those worst-case columns; ‚âà 2.1œÉ on the doubled-twin columns (Surgical, DRY). **False-positive rate (hedged per S2):** the independent-binomial estimate is ‚âà 12% per lens per run, ‚âà 40% per-run-any-lens ‚Äî but this assumes independence across arms, columns, and runs, none of which hold cleanly. Actual FPR is unknown without empirical calibration; treat these as best-effort estimates only. A follow-up calibration task (N=10 baseline-vs-baseline re-runs) will produce a measured FPR; until then, trust threshold-tripping signals as "look at it," not as statistical drift confirmations. Tradeoffs of alternatives considered: (a) Wilson confidence intervals would be more principled but add code complexity and require explaining a non-linear gate to authors; (b) raising `trials` from 5 to 10 specifically for real-PR fixtures + synthetic twins (‚âà 25 extra dispatches per run) would shrink worst-case swing stdev to ‚âà 0.316 and permit a 0.5 tolerance with similar FPR ‚Äî deferred to a follow-up once the real-PR pattern beds in. **Acknowledged miss:** real reviewer drift that produces a 0.4‚Äì0.7 swing will not flag at n=5 with the 0.7 tolerance. The gate's purpose is "loud trip-wire," not statistical hypothesis testing.
- **Drift Delta sign semantics (SP5).** Positive Drift Delta (`synth_pass_rate > real_pass_rate`) is the **expected** drift signature ‚Äî the reviewer handles scrubbed-clean synthetic noise better than realistic noise. Negative Drift Delta (`real_pass_rate > synth_pass_rate`) is **anomalous** and indicates the real-PR fixture is too easy (e.g., the injection site is more conspicuous than the synthetic twin's pattern, or the synthetic twin has a quirk the reviewer mishandles). Negative deltas at baseline-write time should flag the real-PR fixture for re-authoring before the baseline is committed ‚Äî **except on columns whose real-PR contributors include the mixed fixture (S4 exemption)**. Mixed-fixture asymmetry (the mixed fixture is easy on one of its two lens axes but hard on the other, plausibly pulling that one column's real rate up into negative-delta territory while behaving correctly on the other column) is a structural property of the mixed-fixture topology, not a re-author signal. The negative-delta warning fires only on single-axis real-PR fixtures (Harness ¬ß2(h)).
- **`_LENS_RE` substring fragility on real-PR diffs.** Real diffs may contain `Lens:` substrings unrelated to the temper lens vocabulary (Haskell idioms, camera code, ML docs) and confuse downstream parsing. **Non-goal:** hardening `_LENS_RE` against real-PR collisions is deferred. Real-PR fixture authors must verify during fixture authoring that the reviewer's output on their slice does not contain stray `Lens:` substrings; if it does, the fixture is rejected or the slice is re-trimmed.

### Dispatch Budget Envelope (R4 SP1)

Aggregating across AC8a baseline iteration and Test Plan ¬ß7 calibration:

- **Baseline-write rebaseline:** ~750 dispatches worst case (10 cycles √ó 75 dispatches per cycle, where one cycle = 15 fixtures √ó 5 trials); ~225 typical (3 cycles).
- **Per-fixture calibration cycle (synthetic-split, Test Plan ¬ß7 ‚Äî canonical derivation).** Test Plan ¬ß7 is the canonical source: 10 fixtures √ó 20 trials per run √ó 2 runs (pre-split + post-split) = **400 dispatches typical (happy path)**; worst case loops up to **~6200 dispatches** (31 runs total = 1 pre-split + up to 3 re-authoring runs √ó 10 fixtures, each run dispatching all 10 fixtures at n=20 = 200 dispatches/run √ó 31 ‚âà 6200). - **Steady-state:** `--compare-baseline` runs after #290 lands cost ~125 dispatches (25 trials √ó 5 fixtures default), not the setup figure. The setup envelope is one-time cost amortized over every subsequent comparison run.
